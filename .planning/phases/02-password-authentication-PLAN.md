# PLAN: Внедрение традиционной авторизации (Логин/Пароль)

## 🎯 Цели и Границы

Добавить опциональную аутентификацию по паролю в SMMplan, сохранив при этом 100% совместимость с текущим Magic-Link беспарольным входом и гостевым оформлением заказов.

---

## 🏗️ 1. Архитектура и Безопасность

1. **Prisma Миграция:** 
   Добавить `passwordHash String?` в модель `User`. Опциональность поля гарантирует, что существующие пользователи и новые гостевые заказы продолжат работать без ошибок.
2. **Хэширование паролей:**
   Используем встроенный в Node.js модуль `crypto.scrypt` с индивидуальной солью (salt) длиной 16 байт и параметрами подбора:
   * `keylen: 64` (размер результирующего хэша)
   * `N: 16384` (CPU/memory cost)
   * `r: 8` (block size)
   * `p: 1` (parallelization)
   * Это стандарт OWASP, который не требует сторонних тяжелых C++ библиотек (типа `bcrypt`), которые часто падают при сборке в Docker и Windows.
3. **Безопасность сессий:**
   Авторизация по паролю создает стандартную JWT сессию через `createSession` точно так же, как и Magic-Link.

## 🛠️ 2. Пошаговые задачи реализации

### Шаг 2.1: База данных
- Изменить схему `prisma/schema.prisma`: добавить `passwordHash String?`.
- Сгенерировать и применить миграцию с помощью безопасного Prisma клиента (`npx prisma migrate dev --name add_user_password_hash`).

### Шаг 2.2: Вспомогательный класс для хэширования (`src/lib/auth/password.ts`)
- Создать файл `src/lib/auth/password.ts` со следующими чистыми методами:
  - `hashPassword(password: string): Promise<string>`
  - `verifyPassword(password: string, hash: string): Promise<boolean>`

### Шаг 2.3: Server Actions для Управления Паролем (`src/actions/auth/password-settings.ts`)
- Реализовать `setPasswordAction(formData: FormData)`:
  - Проверить активную сессию (`verifySession`).
  - Проверить Zod-схему (минимум 8 символов).
  - Сгенерировать хэш и обновить поле `passwordHash` в БД.
- Реализовать `changePasswordAction(formData: FormData)`:
  - Проверить активную сессию.
  - Проверить текущий пароль (verify с существующим хэшем).
  - Сохранить новый пароль.

### Шаг 2.4: Server Action для Входа (`src/actions/auth/password-login.ts`)
- Реализовать `loginWithPasswordAction(prevState: any, formData: FormData)`:
  - Проверить лимиты скорости (Rate Limiting) по IP и email.
  - Найти пользователя по email. Если у него нет пароля (поле `passwordHash === null`), вернуть ошибку: *"Для вашего аккаунта не установлен пароль. Пожалуйста, воспользуйтесь входом по ссылке."*
  - Сравнить хэш.
  - При успехе создать сессию `createSession` и редиректнуть в дашборд.

### Шаг 2.5: UI Личного Кабинета (`src/app/dashboard/settings/page.tsx`)
- Добавить премиальный блок **«Пароль и безопасность»** ниже основных настроек.
- Интегрировать форму изменения пароля с защитными лоадерами, поддержкой Zod-валидации и реактивными уведомлениями об успешном изменении.

### Шаг 2.6: UI Страницы Входа (`src/app/login/page.tsx`)
- Переработать форму входа с использованием вкладок HeroUI v3 (`<Tabs>`, `<Tab>`):
  - **Вкладка 1:** «Вход по ссылке» (Magic-Link) — текущее проверенное решение.
  - **Вкладка 2:** «Вход по паролю» — новая форма с полями Email и Пароль.

## 🧪 3. План верификации (UAT)

1. **Регистрация нового гостя и логин:**
   - Оформить гостевой заказ, дождаться создания аккаунта.
   - Зайти в `/dashboard/settings`, увидеть карточку «Установить пароль».
   - Установить пароль.
2. **Вход по паролю:**
   - Разлогиниться.
   - Зайти на `/login`, выбрать вкладку «Вход по паролю».
   - Ввести неверный пароль -> увидеть ошибку.
   - Ввести верный пароль -> успешно войти в ЛК.
3. **Изменение пароля:**
   - Перейти в `/dashboard/settings`, увидеть карточку «Сменить пароль».
   - Попробовать сменить с неверным текущим паролем -> увидеть ошибку.
   - Успешно сменить пароль -> разлогиниться -> войти по новому паролю.
4. **Типы и сборка:**
   - Запустить `npx tsc --noEmit` для проверки типов.
   - Запустить `npm run dev` / `npm run build` для подтверждения успешной компиляции всего проекта.
