# План E2E-тестирования Административной Панели Smmplan (Playwright)

## 📌 Цель тестирования
Гарантировать, что обновления кодовой базы не сломают критические бизнес-процессы администраторов (изменение балансов, управление заказами, финансовое ценообразование). Тесты будут эмулировать реальное поведение живого человека в браузере (клики, ожидание загрузки, заполнение форм).

---

## 🎭 Роли и Авторизация
*   **ADMIN / OWNER:** Главная роль для всех E2E сценариев. 
*   **Bypass-механика:** В тестах мы будем инжектить моковый JWT/Session cookie, чтобы не проходить `login` форму при каждом прогоне, экономя время.

---

## 🧪 Сценарии (Test Suites)

### Suite 1: Управление Пользователями (CRM Flow)
**Файл:** `tests/e2e/admin/users.spec.ts`
1.  **Поиск юзера:** Зайти в `/admin/clients`, ввести email в строку поиска, дождаться debounce (300ms) и убедиться, что список отфильтровался.
2.  **Изменение баланса (Пополнение):**
    *   Клик по юзеру -> Открытие сайдбара профиля.
    *   Нажать "Изменить баланс" -> Выбрать тип "ADD", ввести сумму `500.00` и причину `TEST_REWARD`.
    *   **Ожидаемый результат:** Баланс юзера увеличился на 500₽, появилась зеленая Toaster-уведомление, в таблице транзакций появилась запись `TEST_REWARD`.
3.  **Блокировка (Ban):**
    *   Нажать "Заблокировать" -> Подтвердить в модалке.
    *   **Ожидаемый результат:** Статус изменился на BANNED, юзер больше не может логиниться.

### Suite 2: Ценообразование и Каталог (Catalog Flow)
**Файл:** `tests/e2e/admin/catalog.spec.ts`
1.  **Массовая установка маржи:**
    *   Зайти в `/admin/catalog`.
    *   Выделить 3 любые услуги чекбоксами.
    *   В появившемся Batch Action Bar ввести множитель `2.5` и нажать "Применить".
    *   **Ожидаемый результат:** У всех 3 услуг конечная цена (RUB) пересчиталась и сохранилась в БД, маржа стала `x2.50`.
2.  **Защита от убытков (Safety Floor):**
    *   Попытаться установить розничную цену так, чтобы маржа стала ниже порога безубыточности (например, меньше `x1.35`).
    *   Убрать фокус с поля.
    *   **Ожидаемый результат:** UI сбрасывает значение обратно, появляется пульсирующий бэйдж "УБЫТОК" и Toaster-ошибка.

### Suite 3: Маршрутизация Заказов (Order Hub)
**Файл:** `tests/e2e/admin/orders.spec.ts`
1.  **Omni-search:** Ввести ID заказа или кусок ссылки (например, `t.me/durov`).
    *   **Ожидаемый результат:** Таблица моментально отображает 1 релевантный заказ.
2.  **Частичный возврат (Partial Refund):**
    *   Выбрать заказ в статусе `PARTIAL`.
    *   Нажать Action -> Возврат. Указать "Остаток: 500 из 1000".
    *   **Ожидаемый результат:** Заказ получает статус `CANCELED`, 50% суммы возвращается на баланс пользователя (проверяется по транзакции в профиле юзера).

### Suite 4: Финансы и Биллинг (Ledger)
**Файл:** `tests/e2e/admin/finance.spec.ts`
1.  **Escrow / Карантин тест (Тестовый сценарий):**
    *   Убедиться, что при создании "фейкового" заказа через API с подменным параметром стоимости, Дашборд зафиксирует расхождение (Ledger Mismatch).

### Suite 5: Тикетная Система (Support)
**Файл:** `tests/e2e/admin/tickets.spec.ts`
1.  **Ответ пользователю (Optimistic UI):**
    *   Зайти в `/admin/tickets`. Открыть любой `OPEN` тикет.
    *   Ввести сообщение "Hello from Playwright", прикрепить тестовый `.jpg` файл и нажать "Отправить".
    *   **Ожидаемый результат:** Сообщение мгновенно появилось с Loader-ом, через секунду статус сменился на "доставлено". Тикет изменил статус на `PENDING` (Ожидает ответа клиента).

## 🛠 Архитектура Инфраструктуры Тестов
1.  **База Данных:** Тесты будут работать с изолированной базой `test_smmplan.db` (или выделенной схемой PostgreSQL).
2.  **Seed Data:** Перед каждым тестом (`beforeAll`) мы будем запускать скрипт, создающий 10 пользователей, 50 заказов и 1 тикет, чтобы гарантировать чистоту среды.
3.  **Trace Viewer:** При падении теста Playwright автоматически запишет видео и сохранит DOM-дерево для дебага в CI/CD (GitHub Actions).

---

## 🚦 План внедрения
1. Инициализация `npm init playwright@latest` в корне проекта.
2. Настройка `playwright.config.ts` с учетом `test` базы данных.
3. Разработка мокового `auth` стейта (session injection).
4. Написание первого сценария `users.spec.ts` (Изменение баланса).
5. Добавление команды `test:e2e` в `package.json`.
