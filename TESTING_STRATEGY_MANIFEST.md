# Манифест Стратегии Тестирования Smmplan (Enterprise 2026)

Этот документ описывает ультимативную многоуровневую методологию тестирования (Quality Engineering), разработанную командой Senior/Staff инженеров. Цель — достижение **100% уверенности** в коде, отказоустойчивости финансового ядра (FOA), стабильности фронтенда (React 19 / Next.js 16) и полном отсутствии скрытых дефектов и дубликатов. 

Тестирование в 2026 году — это не просто "покрытие строчек кода", это **проактивное управление рисками (Risk-Based Assurance)**.

---

## Уровень 0: Проактивная аналитика и Дизайн (Continuous Risk)

### 1. Тестирование Премортем (Premortem Testing)
* **Концепция:** "Представь, что проект рухнул. Почему это произошло?". Мы анализируем архитектуру до написания кода и пишем тесты на основе наихудших сценариев.
* **Применимо к модулям:** Архитектура FOA (Failure-Oriented Architecture), Модели транзакций Prisma, Дизайн интеграции с провайдерами (VexBoost, Yookassa).
* **Как применять:** 
  - Формирование документа "Threat Model" для каждого нового модуля.
  - Написание тестов-спецификаций, проверяющих, что система НЕ делает того, чего не должна (например: "пользователь не может увести баланс в минус при параллельных запросах").

### 2. Мутационное тестирование (Mutation Testing)
* **Концепция:** Тестирование самих тестов. Инструмент (например, `Stryker`) искусственно вносит ошибки (мутации) в ваш рабочий код (например, меняет `+` на `-`, `>` на `>=`). Если тесты при этом "зеленые" (проходят) — значит тесты плохие, они пропускают баги.
* **Применимо к модулям:** Критическая бизнес-логика (расчет цен, скидок, конвертация валют, `WalletOps`).
* **Как применять:** Внедрение `Stryker Mutator` для папки `src/services/financial/` и `src/utils/pricing.ts`. Цель: 100% Mutation Score (ни один выживший мутант).

## Уровень 1: Backend, Финансы и Базы Данных (Core Integrity)

### 3. Property-Based Testing (Генеративное тестирование)
* **Концепция:** Вместо написания хардкодных моков (ввод "А" -> выход "Б"), мы описываем *свойства* (инварианты), которые всегда должны быть истинны, и прогоняем через инструмент (например, `fast-check`) 10 000 случайных комбинаций данных.
* **Применимо к модулям:** Ценообразование (`PricingService`), расчет маржи и комиссии, калькуляторы скидок, Drip-Feed дробление.
* **Пример свойства:** "При любой цене и любой наценке, итоговая цена для клиента никогда не может быть ниже себестоимости провайдера".

### 4. Тестирование границ транзакций (Transaction Boundary Testing)
* **Концепция:** Тестирование того, как система ведет себя при разрывах коннектов или частичных фейлах прямо посреди транзакции БД.
* **Применимо к модулям:** `WalletOps` (начисление баланса), `OrderProcessor` (списание + создание заказа), Webhooks YooKassa.
* **Как применять:** Интеграционные тесты с реальной базой (как `escrow-flow.test.ts`), где искусственно мокается ошибка внутри `db.$transaction`, и мы проверяем, что откаты (Rollback) сработали корректно, а деньги не зависли.

### 5. Хаос-Инжиниринг (Chaos Engineering & Fault Injection)
* **Концепция:** Искусственный вброс сбоев в работающую систему (отключение сети, падение Redis, задержки API провайдеров до 30 секунд) для проверки самовосстановления системы.
* **Применимо к модулям:** Воркеры BullMQ (`OrderProcessor`, `SyncProcessor`), Взаимодействие со сторонними API (провайдеры).
* **Как применять:** Инъекция `NetworkError` и `Timeout` в тесты воркеров. Проверка того, что заказы корректно уходят в Dead Letter Queue (DLQ), а не исчезают, и что `QuarantineGuard` включает cooldown для мертвых провайдеров.

## Уровень 2: Безопасность и "Красная команда" (Red Team)

### 6. Red-Team & Penetration Testing (Automated)
* **Концепция:** Автоматизированные скрипты, пытающиеся "взломать" систему через известные векторы атак (Race Conditions, IDOR, Negative Balance Exploits).
* **Применимо к модулям:** Checkout (оформление заказа), Auth & Session Guards, Admin Panel API.
* **Как применять:**
  - **Concurrency Tests (Race Conditions):** Запуск 50 параллельных транзакций списания при балансе, достаточном только на 1 транзакцию. Ожидание: 1 успех, 49 отказов (Serializable Isolation).
  - **IDOR (Insecure Direct Object Reference):** Попытки пользователя A отменить заказ пользователя B или получить чужой инвойс.

## Уровень 3: Frontend и Admin Panel (UI/UX Confidence)

### 7. End-to-End (E2E) тестирование критических путей
* **Концепция:** Робот открывает реальный браузер и проходит путь пользователя от регистрации до успешного заказа, кликая по кнопкам.
* **Инструмент:** `Playwright` или `Cypress`.
* **Применимо к модулям:** Весь клиентский Dashboard (`/dashboard/new-order`), Оплата (через мок YooKassa), Авторизация.
* **Как применять:** Настроить CI/CD пайплайн, который поднимает весь стек (Next.js + Postgres + Redis) и прогоняет "Happy Paths" перед каждым релизом.

### 8. Visual Regression Testing (Визуальное тестирование)
* **Концепция:** Система делает скриншоты всех страниц сайта и Админ панели в разных разрешениях (Mobile, Desktop) и попиксельно сравнивает их с эталоном. Если верстка "поехала" из-за изменения Tailwind-класса — тест падает.
* **Инструмент:** `Chromatic` (Storybook) или `Percy`.
* **Применимо к модулям:** UI компоненты (кнопки, модалки), графики аналитики в Админке, сложные таблицы (TanStack Table).

### 9. Accessibility (a11y) & DOM Integrity Testing
* **Концепция:** Проверка того, что UI не только рендерится, но и логически правилен (отсутствуют мертвые/невидимые кнопки, правильные ARIA-атрибуты, фокусы).
* **Инструмент:** `Axe` + Playwright.
* **Применимо к модулям:** Формы создания услуг в админке, навигация.

## Уровень 4: Чистота кода и Архитектуры (Static & Structural)

### 10. Глубокий статический анализ (Deep Static Analysis)
* **Концепция:** Поиск мертвого кода, циклических зависимостей и дубликатов логики без запуска самого кода.
* **Инструмент:** `Knip` (поиск неиспользуемых экспортов и файлов), `SonarQube` (анализ дублирования и Code Smells), `ESLint Flat Config` (с жесткими правилами архитектуры, например "no use client in pages").
* **Применимо к модулям:** Вся кодовая база.
* **Как применять:** Запускается на каждый коммит (Pre-commit hook). Если найден неиспользуемый файл — коммит блокируется.

## 🗺️ Дорожная карта внедрения для Smmplan (Пошаговый план)

Чтобы достичь 100% покрытия и уверенности, команда агентов рекомендует следующий порядок внедрения:

1. **Фаза 1: Финансовый монолит (Недели 1-2)**
   - Настроить **Property-Based Testing (Fast-Check)** для всех калькуляций цен.
   - Завершить интеграционные тесты для **Transaction Boundaries** (убедиться, что `WalletOps` непробиваем).
   - Внедрить скрипт **Race Condition / Red Team** проверки (50+ параллельных запросов на Checkout).

2. **Фаза 2: Отказоустойчивость инфраструктуры (Недели 3-4)**
   - Полное покрытие **Chaos Engineering** для воркеров BullMQ (имитация падений API провайдеров и таймаутов БД).
   - Внедрение **Premortem-моделирования** для нового функционала (например, перед добавлением крипто-шлюзов).

3. **Фаза 3: Визуальная и E2E стабильность (Недели 5-6)**
   - Настройка **Playwright E2E** для флоу клиента (пополнение баланса -> заказ -> статус) и админа (создание услуги -> просмотр логов).
   - Подключение **Visual Regression (Chromatic)** для защиты дизайн-системы Tailwind v4.

4. **Фаза 4: Мутации и Чистота (Неделя 7)**
   - Интеграция **Stryker Mutation Testing** на ядро сервисов для выявления "плохих" тестов.
   - Подключение CI/CD пайплайна с жестким Quality Gate (100% прохождение тестов, 0 дубликатов по Sonar, 0 мертвых файлов по Knip) для допуска в `main`.
