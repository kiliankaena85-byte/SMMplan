# UX/UI Аудит Юзабилити: Вкладка "Каталог Услуг"

## ВВОДНЫЕ ДАННЫЕ О ВКЛАДКЕ
- **Название вкладки:** Каталог Услуг
- **Целевая аудитория (Роли):** Owner (Владелец), CEO. (Для Support вкладка полностью скрыта).
- **Ключевые функции:** Навигация по тысячам услуг, ценообразование, скрытие отработавших (или проблемных) позиций, контроль уровня маржи.

---

## 🔬 АНАЛИЗ ЧЕРЕЗ 5 UX-ФИЛЬТРОВ

### 1. Контекст Ролей (Role Context)
- **Цель Владельца:** Быстро понять, зарабатывает ли он деньги на популярных услугах, оперативно скорректировать цены при колебаниях курса $ или изменениях у провайдера.
- **Проблема:** Сейчас он видит пустую техническую таблицу сомнительной ценности без акцента на прибыли в рублях.
- **Решение:** Вывести на первый план рубрики "Топ продаж" и "Убыточные", чтобыOwner сразу видел, где теряются деньги.

### 2. Плотность Данных (Data Density & Readability)
- **Проблема:** В таблице сотни услуг, а ширина колонок пожирает пространство. Название услуги сливается с бейджами. Нет четких якорей для взгляда.
- **Решение (Идеальная композиция):** 
  - Слева внедряем стационарное узкое "Дерево категорий" (Instagram, TikTok) с бейджами количества услуг.
  - В самой таблице колонка ID должна быть микроскопической моноширинной. Название — сжато до 40% ширины, возможно в две строки. Оставляем *только* Закуп (в $), Нашу Цену (в ₽) и Чекбокс Активности. Все второстепенное (Лимиты, Провайдер) убираем в tooltip или в `<DropdownMenu>`.

### 3. Трение действий (Action Friction)
- **Проблема:** Невозможно задать цену в формате "Хочу продавать за 50₽". Пользователь должен сидеть с калькулятором и подбирать десятичный "Multiplier", скажем 3.7. Это издевательство над B2B.
- **Решение:** Строка таблицы становится инлайн-формой. Пользователь видит два инпута (в одной ячейке): один — маржа (%), второй — цена продажи (₽). Изменяешь один — второй меняется на лету. При потере фокуса (`onBlur`) строка тихо сохраняется с зеленым "Check" без перезагрузки страницы (Optimistic UI / Action State).

### 4. Предотвращение ошибок (Error Prevention)
- **Проблема:** Существующая кнопка "Применить ко всем" деструктивна и находится сверху без сильного сдерживающего фактора. Введенный multiplier 0.5 обанкротит сервис.
- **Решение:** Цвет цены продажи становится *автоматическим индикатором-светофором*. Если после пересчета курса/введенных данных `markup` падает ниже `SAFETY_FLOOR_MARKUP` — ячейка цены окрашивается в тревожный красный цвет с иконкой ⚠️, а строка визуально "кровоточит" (красный border). Сохранение убыточной цены требует двойного клика (или выскочит warning).

### 5. Массовые операции (Batch Processing)
- **Проблема:** 3000 услуг. Изменение цен вручную для каждой позиции в категории Instagram займет часы. 
- **Решение:** Возле заголовка текущей категории появляется мощный ActionBar (закрепляющийся при скролле). "Применить +30% к текущей категории", либо мульти-select через чекбоксы (Gmail-style) и изменение цены пачкой через модальное окно.

## 🏗 ВЫВОД: АРХИТЕКТУРНОЕ ТЗ

### A) Архитектура UI компонента
Интерфейс строится по паттерну **Two-Pane Workspace**:
* **Sidebar (Left, 250px):**
  - Search Input (быстрый текстовый фильтр).
  - List of Categories: `Все услуги [3118]`, `⚠️ Убыточные [4]`, `Instagram [856]`, `Telegram [234]`...
* **Main Area (Right, flex-1):**
  - **Header:** Название выбранной категории, кнопка массовых действий `Bulk Actions ▾`.
  - **Data-Grid (Таблица):** Заголовки `ID | Услуга | Провайдер (Logo) | Закуп | Розница (₽) | Статус`.
  - **Cell [Розница]:** Мощный компонент с двумя полями: `[ Margin % ]` и `[ Price ₽ ]`.

### B) Техническое ТЗ для Frontend-разработчика
1. **Layout:** Использовать Grid `grid-cols-[250px_1fr]` для страницы Каталога.
2. **HeroUI Components:** 
   - Активно применять `Tabs` (вертикальные) или `Listbox` для левого категорийного сайдбара.
   - Использовать `Table` с `selectionMode="multiple"` для встроенного Batch Processing.
   - `DropdownMenu` для скрытых действий над каждой услугой (Скрыть из прайса, Пометить как ТОП).
3. **Logic:** Написать клиентский хук `usePricingForm(baseRate, defaultMarkup)`. Он должен обеспечивать two-way binding. Если юзер вводит цену в ₽ (150₽), хук пересчитывает `%` и вызывает `useActionState` или `useOptimistic` для Server Action без тормозов.
4. **Action State:** Встроить бекграунд-подсветку: `<td className="data-[state=error]:bg-rose-50 ...">`
