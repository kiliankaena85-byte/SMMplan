# Рефакторинг UX/UI Административной Панели Smmplan (Deep Usability Audit & Plan)

Этот документ является глубоким анализом текущих UX-проблем админ-панели и планом внедрения новой, профессиональной Data-Grid архитектуры, заточенной под B2B.

## 1. UX Аудит текущих проблем (As-Is)

Я проанализировал текущий код (`admin/layout.tsx`, `catalog/page.tsx`, `client-table.tsx`) и полностью согласен с вами. Текущая реализация админки имеет критические изъяны в юзабилити:

### ❌ Проблема 1: Изолированность категорий и слепой скроллинг
- **Как сейчас:** Каталог выводит все 3000+ услуг одним плоским списком с пагинацией по 50 штук. Категории (`/admin/catalog/categories`) вынесены на отдельную, никак не связанную страницу. 
- **Боль юзера:** Чтобы найти услуги конкретной социальной сети (например, только Instagram), нужно либо листать десятки страниц, либо надеяться на текстовый поиск. Это не "управление каталогом", это слепой перебор данных.

### ❌ Проблема 2: Когнитивная нагрузка при ценообразовании (Markup UX)
- **Как сейчас:** Присутствует кнопка-убийца "Bulk маржа: Применить ко всем", которая тупо умножает всё на одно число. Для отдельных услуг нужно вводить "множитель" (например, `3.2`), а не итоговую цену продажи в рублях или процент маржинальности. Отсутствует кнопка "Сохранить" отдело у инпута, что делает логику сохранения неочевидной (авто-сохранение или нужен Enter?).
- **Боль юзера:** Человек мыслит категориями прибыли ("хочу продавать это за 150₽" или "хочу маржу 60%"), а не абстрактными математическими множителями относительно долларовой цены провайдера.

### ❌ Проблема 3: Плоская навигация без иерархии
- **Как сейчас:** В `AdminSidebar` свалены в одну кучу 11 элементов (`Дашборд, Клиенты, Каталог, Провайдеры, Страницы...`). 
- **Боль юзера:** Нет визуального разделения между "Ежедневной операционкой" (Заказы, Тикеты) и "Настройками ядра" (Провайдеры, Формирование цен, Страницы). Глаз цепляется за всё подряд.

## 2. Предлагаемая UX-Архитектура (To-Be)

Мы перейдем к стандарту **Command Center** (плотные данные, высокая скорость работы, фокус на Desktop, четкая иерархия).

### 2.1 Новая навигация (Sidebar Grouping)
Сайдбар будет разбит на смысловые секции:
- **📊 Операционка:** Дашборд, Заказы (Orders / Refills склеим визуально), Тикеты, Клиенты.
- **💰 Финансы & Продажи:** Финансы (Billing), Маркетинг.
- **🛠 Ядро & Инфраструктура:** Каталог Услуг, Провайдеры API.
- **⚙️ Настройки:** Роли & Доступы, Системные Настройки.

### 2.2 Модуль "Каталог" (Ultimate Usability)
Переписываем страницу `/admin/catalog`:
- **Умные табы (Tabs):** Сверху (или слева 2-м уровнем) появляется список Категорий (Instagram, TikTok, Telegram). При клике загружаются только услуги этой категории.
- **Human-Readable Ценообразование:** В таблице услуг мы делаем инпут **не для множителя**, а для **конечной цены в рублях** (или % маржи). Пользователь вписывает: *Закуп: 10₽. Продажа: [ 50 ] ₽*. А система *сама под капотом высчитывает нужный multiplier (x5.0)* и сохраняет в БД через Server Action на `onBlur` с красивой зеленой галочкой.
- **Массовые действия (Bulk Actions 2.0):** Убираем опасную кнопку "Для всех". Добавляем чекбоксы к таблице. Можно выделить 10 услуг из категории Telegram и нажать "Установить маржу +40% для выбранных".

### 2.3 Role-Based Access Control (RBAC)
- **SUPPORT:** Видит в сайдбаре только: Дашборд (урезанный), Клиенты (без балансов), Заказы (без себестоимости провайдера, только цена продажи), Тикеты. Остальные роуты жестко перекрыты через `verifySession` и middleware.
- **CEO:** Видит всё, кроме раздела "Провайдеры" и "Настройки". 
- **OWNER:** Бог системы, видит всё.

## 3. План реализации (Technical Execution Steps)

1. **Рефакторинг Меню (Sidebar & Layout):** Отредактировать массивы ролей и группировку в `src/app/admin/layout.tsx` и `src/components/admin/sidebar.tsx`. Добавить секционирование (Headers: Операционка, Финансы, Настройки).
2. **Создание Каталога с категориями (Tabs):** 
   - Обновить `page.tsx` в каталоге: загружать левое подменю с `category.name` и `category._count.services`.
   - Настроить URL state (`?category=id`), чтобы Server Component `CatalogTable` рендерил только нужные услуги.
3. **Рефакторинг таблицы цен (Pricing UX):** 
   - Переработать `client-table.tsx`: вместо абстрактного `markup`, выводить инпут конечной цены `price (₽)`. Написать функцию-инвертер: если вводим цену 150₽, она рассчитывает `markup = 150 / (rate * USD_TO_RUB)`.
   - Добавить toast-уведомления об успешном сохранении строки.
4. **Внедрение RBAC:**
   - Внедрить утилиту `requireRole(roles[])` в критические Server Actions.
   - Очистить интерфейсы (убрать закупочные цены для Support роли).
